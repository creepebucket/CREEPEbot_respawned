# Repository Guidelines

5. 坚持 KISS 原则，方案必须简单直接。
6. 任何重复的逻辑或方法都要抽到父类，避免在子类里堆叠。
7. 排查 bug 必须定位根本原因，修复源头，而不是随便打补丁。
8. 禁写防御性或“兜底”代码，那样既不解决问题也增加调试成本。
9. 不得改动我已有的注释内容。
10. 不得把本文件的内容抄进代码注释。
11. 如果修 bug 无法在 3 行以内完成，就不要直接改代码，只需说明原因并给出建议。
12. 修 bug 时要顺着我现在的设计逻辑来，别动不动就推翻或重写整体逻辑，先按现有实现思路去定位并解决。
14. 代码保持平铺式：按执行顺序直接写在当前上下文，流程清晰、调用栈浅。
15. 所有需求都需要查看已有轮子和工具实现，不要重复造轮子。
16. 默认信任我构造的状态，必要检查提前放在固定位置，别到处加防御性判断或兜底逻辑。
18. 修 bug 时只修我指出的 bug, 其他潜在问题只允许提出, 不准修。
19. 不要写完整的包名, 使用import语句引入。
21. 不允许查看图片, 这对工作没有任何帮助, 相信我给的数据
22. 对于我要求的重构, 不允许"顺手" "修"其他代码内容, 除非特殊要求"这一块全部重构", 否则只允许提建议
23. 写代码不允许超过 3个 tab 缩进(定义类的不算), 也就是在方法里只允许两层 if/for/while嵌套
24. 不允许用分号分隔本该是多行的代码强行到一行
25. 不允许写只使用一次的函数和变量, 合并到调用点
